对OS的初期探索

Lab1

  • C语言中include是有先后的，如果一个头文件中引用了另一个头文件中的定义或内容，则被引用的头文件必须在引用的那个之前出现
  • 管道也有文件偏移量，输出后无法再读取

    Lec1

  • 每个进程都有自己独立的文件描述符空间
  • read(0,buf,sizeof(buf)) write同理
    第一个参数是文件描述符，指向一个之前打开的文件。Shell会确保默认情况下，当一个程序启动时，文件描述符0连接到console的输入，文件描述符1连接到了console的输出;第二个参数是指向某段内存的指针;第三个参数是代码想读取的最大长度
  • open(“output.txt”,O_WRONLY | O_CREATE)
    第二个参数是一些标志位，用来告诉open系统调用在内核中的实现：我们将要创建并写入一个文件。
    open会返回当前进程未使用的最小文件描述符序号
  • 子父进程有相同内容但各自私有的文件描述符表。但是每个基础文件偏移量在父文件和子文件之间是共享的
  • 不可能让子进程等待父进程退出。
  • 引用文件的每个文件描述符都有一个与之关联的偏移量。read从当前文件偏移量开始读取数据，然后将该偏移量前进所读取的字节数：（也就是说）后续读取将返回第一次读取返回的字节之后的字节。
  • 按照惯例，进程从文件描述符0读取（标准输入），将输出写入文件描述符1（标准输出），并将错误消息写入文件描述符2（标准错误）。
  • dup允许shell执行这样的命令：ls existing-file non-existing-file > tmp1 2>&1。2>&1告诉shell给命令的文件描述符2是描述符1的副本。现有文件的名称和不存在文件的错误信息都会显示在tmp1文件中。

    Lec2

  • 整个操作系统都驻留在内核中，这样所有系统调用的实现都以管理模式运行。这种组织被称为宏内核（monolithic kernel）
  • 最大限度地减少在管理模式下运行的操作系统代码量，并在用户模式下执行大部分操作系统。这种内核组织被称为微内核（microkernel）
  • 作为进程运行的操作系统服务被称为服务器\
  • Client/Server模式：将操作系统中最基本的部分放入内核中，而把操作系统的绝大部分功能都放在微内核外面的一组服务器(进程)中实现。
  • Xv6使用页表（由硬件实现）为每个进程提供自己的地址空间。RISC-V页表将虚拟地址（RISC-V指令操纵的地址）转换（或“映射”）为物理地址（CPU芯片发送到主存储器的地址）
  • 每个进程有两个栈区：一个用户栈区和一个内核栈区（p->kstack）。当进程执行用户指令时，只有它的用户栈在使用，它的内核栈是空的。当进程进入内核（由于系统调用或中断）时，内核代码在进程的内核堆栈上执行；当一个进程在内核中时，它的用户堆栈仍然包含保存的数据，只是不处于活动状态。
  • 进程的线程在主动使用它的用户栈和内核栈之间交替。内核栈是独立的（并且不受用户代码的保护），因此即使一个进程破坏了它的用户栈，内核依然可以正常运行。

    进程状态

  1. 就绪状态，英文称Ready。进程已获得除处理器外的所需资源，只是在等待分配处理器资源，只要分配了处理器进程就可执行。
  • 准备就绪进程可以按多个优先级来划分队列。例如，当一个进程由于时间片用完而进入就绪状态时，排入低优先级队列。当进程由I/O操作完成而进入就绪状态时，排入高优先级队列。

  1. 运行状态，英文称Running。进程占用处理器资源，处于此状态的进程的数目小于等于处理器的数目。在没有其他进程可以执行时(如所有进程都在阻塞状态)，系统通常会自动执行系统的空闲进程。

  2. 阻塞状态，英文称Blocked，系统由于进程等待某种条件（如I/O操作或进程同步），在条件满足之前无法继续执行。该事件发生前即使把处理器资源分配给该进程，该进程也无法进行运行。

    Lec3

  • multiplexing（分时复用）和内存隔离等是操作系统虚拟化一些硬件的特点
  • 操作系统不是直接将CPU提供给应用程序，而是向应用程序提供“进程”，进程抽象了CPU
  • 每一个进程都会有自己独立的page table，每一个进程只能访问出现在自己page table中的物理内存。操作系统会设置page table，使得每一个进程都有不重合的物理内存，这样一个进程就不能访问其他进程的物理内存，因为其他进程的物理内存都不在它的page table中。
  • XV6中存在一个唯一的系统调用接入点，每一次应用程序执行ECALL指令，应用程序都会通过这个接入点进入到内核中
  • 内核有时候也被称为可被信任的计算空间（Trusted Computing Base）

    Monolithic Kernel Design（宏内核）

  • XV6中，所有的操作系统服务都在kernel mode中，这种形式被称为Monolithic Kernel Design（宏内核）
  • 优势：文件系统，虚拟内存，进程管理，这些都是操作系统内实现了特定功能的子模块现在都位于同一个程序中，它们可以紧密的集成在一起，这样的集成提供很好的性能。
  • 劣势：内核中bug增多

    Micro Kernel Design（微内核）

  • 将大部分的操作系统运行在内核之外。
  • 还是会有user mode以及user/kernel mode的边界，但是我们现在会将原来在内核中的其他部分，作为普通的用户程序来运行。
  • 劣势：
  1. 在宏内核中如果一个应用程序需要与文件系统交互，只需要完成1次用户空间<->内核空间的跳转，但微内核的的跳转是宏内核的两倍。
  2. 在一个类似宏内核的紧耦合系统，各个组成部分，例如文件系统和虚拟内存系统，可以很容易的共享page cache。而在微内核中，每个部分之间都很好的隔离开了，这种共享更难实现。进而导致更难在微内核中得到更高的性能。

    Lec4

    页表

  • 对于任何一条带有地址的指令，其中的地址应该认为是虚拟内存地址而不是物理地址。

  • 虚拟内存地址会被转到内存管理单元（MMU，Memory Management Unit）

  • page table保存在内存中，MMU只是会去查看page table

  • 一个叫 satp（Supervisor Address Translation and Protection，监管者地址转换和保护）的 S 模式控制状态寄存器控制了分页系统。

  • 每个进程有不同页表；内核负责在切换进程时写SATP寄存器

    页表工作原理

    页式硬件

  • 每个page创建一条表单条目，所以每一次地址翻译都是针对一个page。而RISC-V中，一个page是4KB，也就是4096Bytes。

  • 对于虚拟内存地址，我们将它划分为两个部分，index和offset，index用来查找page，offset对应的是一个page中的哪个字节。

  • 页表通过逻辑到物理地址的转换给了操作系统控制权，转换的粒度是一个个对齐的物理块（一个物理块包含2^{12}=4096字节），这样的块称为**页面(page)**。注意，一个page在物理内存中是连续的，即物理内存是以page为粒度使用的！

  • Sv39 RISC-V:只使用64位虚拟地址的低39位；而高25位不使用。
    在这种Sv39配置中，RISC-V页表在逻辑上是一个由2^27（134,217,728）个页表条目（Page Table Entries/PTE）组成的数组。每个PTE包含一个44位的物理页码（Physical Page Number/PPN）和一些标志。页式硬件通过使用虚拟地址39位中的前27位索引页表，以找到该虚拟地址对应的一个PTE，然后生成一个56位的物理地址，其前44位来自PTE中的PPN，其后12位来自原始虚拟地址。

  • 在RISC-V中，物理内存地址是56bit。所以物理内存可以大于单个虚拟内存地址空间，但是也最多到2^56。

    多级页表

  • RISC-V中，页表以三级的树型结构存储在物理内存中。27位index被分为L2,L1,L0各9位，从根到叶索引page directory。一个directory是4096Bytes，就跟page的大小是一样的。Directory中的一个条目被称为PTE（Page Table Entry）是64bits，就像寄存器的大小一样，也就是8Bytes。所以一个Directory page有512个条目（对应9位索引）。

  • 所有的page directory传递的都是PPN，对应的物理地址是44bit的PPN加上12bit的0（注，也就是page的起始地址，因为每个page directory都使用一个完整的page，所以直接从page起始地址开始使用就行）。这里要求每个page directory都与物理page对齐

    页表缓存（Translation Lookaside Buffer） TLB

  • 在RISC-V中，清空TLB的指令是sfence_vma

  • 对操作系统不可见

  • 在一个RISC-V芯片中，有多个CPU核，MMU和TLB存在于每一个CPU核里面。对于TLB，RISC-V处理器有L1 cache，L2 Cache，有些cache是根据物理地址索引的，有些cache是根据虚拟地址索引的，由虚拟地址索引的cache位于MMU之前，由物理地址索引的cache位于MMU之后。

    kernel page table

  • 地址0x1000是boot ROM的物理地址，当你对主板上电，主板做的第一件事情就是运行存储在boot ROM中的代码，当boot完成之后，会跳转到地址0x80000000，操作系统需要确保那个地址有一些数据能够接着启动操作系统。

  • 未被映射的Guard page：这个Guard page对应的PTE的Valid 标志位没有设置，这样，如果kernel stack耗尽了，它会溢出到Guard page，但是因为Guard page的PTE中Valid标志位未设置，会导致立即触发page fault，这样的结果好过内存越界之后造成的数据混乱。Guard page不会映射到任何物理内存，它只是占据了虚拟地址空间的一段靠后的地址，不会浪费内存。

  • 虚拟地址到物理地址可以是一对一的映射，一对多映射，多对一映射

  • Xv6为每个进程维护一个页表，用以描述每个进程的用户地址空间。内核维护一个一个描述内核地址空间的页表。（每个进程都有一个进程页表，但内核页表只有一个，所有进程共享）。在内核虚拟地址空间每个进程都有自己的内核栈。

  • 在kernel page table中，有一段Free Memory，它对应了物理内存中的一段地址。
    XV6使用这段free memory来存放用户进程的page table，text和data。如果我们运行了非常多的用户进程，某个时间点我们会耗尽这段内存，这个时候fork或者exec会返回错误。

  • 每个RISC-V CPU都将页表条目缓存在转译后备缓冲器（快表/TLB）中，当xv6更改页表时，它必须告诉CPU使相应的缓存TLB条目无效。如果没有这么做，那么在某个时候TLB可能会使用旧的缓存映射，指向一个在此期间已分配给另一个进程的物理页面，这样会导致一个进程可能能够在其他进程的内存上涂鸦。RISC-V有一个指令sfence.vma，用于刷新当前CPU的TLB。

    TLB击落

    如果操作系统修改了页表，那么这个缓存会变得陈旧而不可用。S 模式添加了另一条指令来解决 这个问题。这条 sfence.vma 会通知处理器，软件可能已经修改了页表，于是处理器可以 相应地刷新转换缓存。它需要两个可选的参数，这样可以缩小缓存刷新的范围。一个位于 rs1，它指示了页表哪个虚址对应的转换被修改了;另一个位于 rs2，它给出了被修改页表 的进程的地址空间标识符(ASID)。如果两者都是 x0，便会刷新整个转换缓存。

    Lec5

  • Return address总是会出现在Stack Frame的第一位

  • a0-a7 共8个寄存器可以保存参数（调用argraw来检索相应的保存的用户寄存器）

  • ra寄存器保存当前函数的返回地址；jal指令暗含覆盖ra寄存器的操作，所以要在调用者的栈帧中保存调用者的返回地址

  • ecall并不会切换page table.所以这意味着，trap处理代码必须存在于每一个user page table中。因为ecall并不会切换page table，我们需要在user page table中的某个地方来执行最初的内核代码。而这个trampoline page，是由内核小心的映射到每一个user page table中，以使得当我们仍然在使用user page table时，内核在一个地方能够执行trap机制的最开始的一些指令。

    从用户空间陷入

    ecall指令作用

  1. 切换mode到S mode
  2. 将用户PC值保存在SEPC（Supervisor Exception Program Counter）
  3. ecall指令使PC跳转到STVEC(S mode r)保存的地址处，即用户地址空间的trampoline page(PTE_U未设置)处开始执行
  • 一台机器总是从内核开始运行的，进入到用户空间的唯一方法是就是执行sret指令。
  • 内核要保存寄存器的原因，是因为内核即将要运行会覆盖这些寄存器的C代码。同时，我们不确定用户程序是否有栈，也有一些编程语言有栈，但是或许它的格式很奇怪，内核并不能理解，所以内核将这些内容保存在属于内核内存的trapframe中，而不是用户内存。

    uservec

  1. 交换sscratch(trapframe va)与a0的值，并将原本a0保存到trapframe中
  2. 保存寄存器，cpu hartid,usertrap指针，
  3. 从tramframe加载kernel page table到satp
  • XV6会将CPU核的编号也就是hartid保存在tp寄存器。
  • kernel/user page table中其他所有的映射都是不同的，只有trampoline page的映射是一样的
    ecall 跳转到trampoline.S中，首先执行其中的uservec函数(ams)，保存寄存器和几个重要参数后，跳转到usertrap.c，第一个在内核中的.c代码。

    usertrap.c

  1. 先将STVEC指向了kernelvec变量，这是内核空间trap处理代码的位置，而不是用户空间trap处理代码的位置
  2. 使用trapframe来保存用户程序计数器(SEPC中)
  3. 中断总是会被RISC-V的trap硬件关闭。设置了kernelvec到stvec后，显式的打开中断。(RISC-V总是在开始设置陷阱时禁用中断，xv6在设置stvec之前不会再次启用中断)
  4. 调用syscall()
  • 所有的系统调用都有一个返回值，比如write会返回实际写入的字节数，而RISC-V上的C代码的习惯是函数的返回值存储于寄存器a0，所以为了模拟函数的返回，我们将返回值存储在trapframe的a0中。之后，当我们返回到用户空间，trapframe中的a0槽位的数值会写到实际的a0寄存器，Shell会认为a0寄存器中的数值是write系统调用的返回值。

    usertarpret.c

  1. 关闭中断(我们关闭中断因为当我们将STVEC更新到指向用户空间的trap处理代码时，我们仍然在内核中执行代码。如果这时发生了一个中断，那么程序执行会走向用户空间的trap处理代码.这会导致内核出错。所以我们这里关闭中断。)
  2. 设置STVEC寄存器指向trampoline代码(sret指令会重新打开中断)
  3. 填入了trapframe的内容
  4. 设置SSTATUS寄存器（控制寄存器）。这个寄存器的SPP bit位控制了sret指令的行为，该bit为0表示下次执行sret的时候，我们想要返回user mode而不是supervisor mode。这个寄存器的SPIE bit位控制了，在执行完sret之后，是否打开中断。因为我们在返回到用户空间之后，我们的确希望打开中断，所以这里将SPIE bit位设置为1。
  5. 从trapframe中取出sepc寄存器的值(在usertrap.c中保存)
  6. 从trapframe恢复SEPC寄存器的值
  7. 在satp寄存器中存储user page table的指针，并作为参数传递给fn(a1中保存)
  • 用户寄存器（User Registers）必须在汇编代码中保存，因为任何需要经过编译器的语言，例如C语言，都不能修改任何用户寄存器。

    userret

  1. 切换page table。user page table也映射了trampoline page，所以程序还能继续执行而不是崩溃
  2. 将SSCRATCH寄存器恢复成保存好的用户的a0寄存器（通过当前的a0寄存器找出存在trapframe中的a0寄存器）(是系统调用返回值)
  3. 恢复寄存器（a0寄存器还是个例外，它现在仍然是指向trapframe的指针）,交换sscratch和a0的值，a0持有syscall返回值
  4. sret指令

    sret干的三件事

  5. 程序会切换回user mode
  6. SEPC寄存器的数值会被拷贝到PC寄存器（程序计数器）
  7. 重新打开中断
    之后从sh.asm的ret处退出(就像开始在user mode中使用封装的syscall进入一样)

    从内核空间陷入

    kernelvec

    Lazy page allocation（Lab5）

    在XV6中，sbrk的实现默认是eager allocation。（在虚拟内存创建时就mappages对应的物理页面）
    主要思想就是在调用sbrk()时只是移动进程中的p->sz这一标志用户应用进程的heap，然后不做其他工作，只在产生page fault的时候去增加内存页面，在sys_sbrk()的时候sbrk会扩展heap的上边界,但不实际去mappages。

对此类page fault的处理思路：在page fault handler中，通过kalloc函数分配一个内存page；初始化这个page内容为0；将这个内存page映射到user page table中；最后重新执行指令。（注意，此时虚拟地址小于旧的p->sz，表示是有效地址；大于旧的小于新的则是待分配地址，大于新的则是无效地址）

需要处理的是一些特殊的错误情况：

  • 释放或copy lazy allocation的部分内存时，即walk==0（叶页表这层都没分配到）或者PTE_V==0（walk不检查叶页表的PTE_V，此时是在叶页表512个之中但是还没分配）
  • sbrk()参数为负
  • 如果某个进程在高于sbrk()分配的任何虚拟内存地址上出现页错误或者在低于用户栈的下方出现页错误，则终止该进程；
  • 进程从sbrk()向系统调用（如read或write）传递有效地址，但尚未分配该地址的内存。（在argadr中处理）
  • 正确处理内存不足：如果在页面错误处理程序中执行kalloc()失败，则终止当前进程。

    zero-fill-on-demand

    将exec申请.bss段的空白页面全部映射到一个空白的物理页面，把启动时的内存开销后移直至对某个页面进行写操作时。
    减小了启动开销、启动时的内存开销，增加了后续page fault带来的开销

Copy On Write Fork（Lab6）

原始的实现：在物理内存中，XV6中的Shell通常会有4个page，当调用fork时，基本上就是创建了4个新的page，并将父进程page的内容拷贝到4个新的子进程的page中。如果父进程较大，那么子进程的拷贝将有大量开销，这些被copy的内存很可能不会被全部利用。

硬件上对COW的支持：page table的64位硬件设施中，bit8/9所代表的RSW(Reserved for supervisor software)位是留给内核支配的。可以设置page的bit8位为1代表这是一个COW page

COW实现思路：只为子进程创建一个页表，用户内存的PTE指向父进程的物理页。COW fork()将父进程和子进程中的所有用户PTE标记为不可写。当任一进程试图写入其中一个COW页时，CPU将强制产生页面错误。内核页面错误处理程序检测到这种情况将为出错进程分配一页物理内存，将原始页复制到新页中，并修改出错进程中的相关PTE指向新的页面，将PTE标记为可写。当页面错误处理程序返回时，用户进程将能够写入其页面副本。

但要注意的是，多个子进程对父进程的同一物理页进行引用的时候，应该进行计数，引用数为零时才能执行用户释放的操作

实验实现注意的问题：

  1. 利用spinlock进行kfree和kalloc操作，在其中对引用计数ref进行增减
  2. ref数据结构可以分配一个cnt数组，大小为内存的物理字节大小除以4096
  3. PTE_W的清除与设置时机

Demand paging

对于exec，在虚拟地址空间中，我们为text和data分配好地址段，但是相应的PTE并不对应任何物理内存page。对于这些PTE，我们只需要将valid bit位设置为0即可。

应用程序是从地址0开始运行。text区域从地址0开始向上增长。位于地址0的指令是会触发第一个page fault的指令，因为我们还没有真正的加载内存。

过程：在某个地方记录了这些page对应的程序文件，我们在page fault handler中需要从程序文件中读取page数据，加载到内存中；之后将内存page映射到page table；最后再重新执行指令。

页面撤回

lazy allocation中，一旦在page fault时出现OOM，那么需要和文件系统置换页面，腾出新的page供page fault使用。

最常用的策略，Least Recently Used，或者叫LRU。会选择non-dirty page进行回收（只被读过，没有被写过）。

一个page只要最近被写过，那么就是dirty的。硬件上在RSW的bit7上设置dirty。当硬件向一个page写入数据，会设置dirty bit。类似的，还有一个Access bit，任何时候一个page被读或者被写了，这个Access bit会被设置。Access bit通常会被定时清除。可以通过Access bit来决定内存page 在LRU中的排名。

mmap

void* mmap(void* start,size_t length,int prot,int flags,int fd,off_t offset);

接收一个虚拟内存地址（VA），长度（len），protection（），一些标志位，一个打开文件的文件描述符，和偏移量（offset）

语义为：从文件描述符对应的文件的偏移量的位置开始，映射长度为len的内容到虚拟内存地址VA，同时我们需要加上一些保护

  1. mmap()必须以PAGE_SIZE为单位进行映射
  2. prot：期望的内存保护标志，不能与文件的打开模式冲突。（被执行、被读取、被写入、不可访问）
  3. flag：指定映射对象的类型，映射选项和映射页是否可以共享。
  4. 实际实现中是lazy分配的，并不会立即将文件内容拷贝到内存中，而是先记录一下这个PTE属于这个文件描述符。相应的信息通常在VMA结构体中保存，VMA全称是Virtual Memory Area。在VMA中我们会记录文件描述符，偏移量等，这些信息用来表示对应的内存虚拟地址的实际内容的位置。

中断

真实内存使用情况

  1. 大部分内存都被使用了（used）。但是大部分内存并没有被应用程序所使用，而是被buff/cache用掉了。
  2. 实际使用的内存数量远小于地址空间的大小
  3. 输入top指令可查看内存使用

中断与系统调用的三个区别

  1. asynchronous（异步）。当硬件生成中断时，Interrupt handler与当前运行的进程在CPU上没有任何关联。但如果是系统调用的话，系统调用发生在运行进程的context下。
  2. concurrency。对于中断来说，CPU和生成中断的设备是并行的在运行。网卡自己独立的处理来自网络的packet，然后在某个时间点产生中断，但是同时，CPU也在运行。所以我们在CPU和设备之间是真正的并行的。
  3. program device。网卡，UART，而这些设备需要被编程。每个设备都有一个编程手册，就像RISC-V有一个包含了指令和寄存器的手册一样。设备的编程手册包含了它有什么样的寄存器，它能执行什么样的操作，在读写控制寄存器的时候，设备会如何响应。

设备驱动

管理设备的代码通常被称为设备驱动，所有的驱动都在内核中。

大部分驱动可分为bottom/top两个部分。前者通常是Interrupt handler。当一个中断送到了CPU，并且CPU设置接收这个中断，CPU会调用相应的Interrupt handler。Interrupt handler不运行在进程的context中。top部分，是用户进程，或者内核的其他部分调用的接口。

RISCV中与中断有关的寄存器

  1. SIE（Supervisor Interrupt Enable）寄存器。这个寄存器中有一个bit（E）专门针对例如UART的外部设备的中断；有一个bit（S）专门针对软件中断，软件中断可能由一个CPU核触发给另一个CPU核；还有一个bit（T）专门针对定时器中断。
  2. SSTATUS（Supervisor Status）寄存器。这个寄存器中有一个bit来打开或者关闭中断。每一个CPU核都有独立的SIE和SSTATUS寄存器，除了通过SIE寄存器来单独控制特定的中断，还可以通过SSTATUS寄存器中的一个bit来控制所有的中断。其中的SIE位控制设备中断是否启用。如果内核清空SIE，RISC-V将推迟设备中断，直到内核重新设置SIE。
  3. SIP（Supervisor Interrupt Pending）寄存器。当发生中断时，处理器可以通过查看这个寄存器知道当前是什么类型的中断。
  4. SCAUSE寄存器。它会表明当前trap状态的原因是中断。
  5. STVEC寄存器，它会保存当trap，page fault或者中断发生时，CPU运行的用户程序的程序计数器，这样才能在稍后恢复程序的运行。

RISCV硬件对中断的支持

所有的设备都连接到处理器上，处理器上是通过Platform Level Interrupt Control，简称PLIC来处理设备中断。中断到达PLIC之后，PLIC会路由这些中断到某一个CPU的核。如果所有的CPU核都正在处理中断，PLIC会保留中断直到有一个CPU核可以用来处理中断。

具体流程为：

  1. PLIC会通知当前有一个待处理的中断
  2. 其中一个CPU核会Claim接收中断，这样PLIC就不会把中断发给其他的CPU处理
  3. CPU核处理完中断之后，CPU会通知PLIC
  4. PLIC将不再保存中断的信息

    对于一个高性能的网卡，如果有大量的包要传入，那么应该用polling。因为大量短时间内的interrupt会使cpu无法接受。对于这样一种快设备，cpu主动轮询是可以经常获取数据的。对于一些精心设计的驱动，它们会在polling和Interrupt之间动态切换。

    DMA（Direct Memory Access）

    UART驱动程序读取UART控制寄存器，一次检索一字节的数据；因为软件驱动数据移动，这种模式被称为程序I/O（Programmed I/O）。程序I/O很简单，但速度太慢，无法在高数据速率下使用。需要高速移动大量数据的设备通常使用直接内存访问（DMA）。DMA设备硬件直接将传入数据写入内存，并从内存中读取传出数据。现代磁盘和网络设备使用DMA。DMA设备的驱动程序将在RAM中准备数据，然后使用对控制寄存器的单次写入来告诉设备处理准备好的数据。

多处理器与锁

锁的特性

  1. 锁可以避免丢失更新
  2. 锁可以打包多个操作，使它们具有原子性
  3. 锁可以维护共享数据结构的不变性

锁带来的挑战

死锁

什么是死锁？

多个进程可以竞争有限数量的资源。当一个进程申请资源时，如果这时没有可用资源，那么这个进程进入等待状态。有时，如果所申请的资源被其他等待进程占有，那么该等待进程有可能再也无法改变状态。这种情况成为死锁。

死锁产生的四个必要条件

  1. 互斥：资源必须处于非共享模式，即一次只有一个进程可以使用。如果另一进程申请该资源，那么必须等待直到该资源被释放为止。
  2. 占有并等待：一个进程至少应该占有一个资源，并等待另一资源，而该资源被其他进程所占有。
  3. 非抢占：资源不能被抢占。只能在持有资源的进程完成任务后，该资源才会被释放。
  4. 环路等待：有一组等待进程 {P0, P1,..., Pn}， P0 等待的资源被 P1 占有，P1 等待的资源被 P2 占有，……，Pn-1 等待的资源被 Pn 占有，Pn 等待的资源被 P0 占有。

预防死锁

  • 锁排序->破坏环路等待：所有代码路径必须以相同的顺序获取锁。全局锁获取顺序的需求意味着锁实际上是每个函数规范的一部分：调用者必须以一种使锁按照约定顺序被获取的方式调用函数。
  • 自旋锁与中断处理程序：如果一个自旋锁被中断处理程序所使用，那么CPU必须保证在启用中断的情况下永远不能持有该锁。Xv6更保守：当CPU获取任何锁时，xv6总是禁用该CPU上的中断。中断仍然可能发生在其他CPU上，此时中断的acquire可以等待线程释放自旋锁；由于不在同一CPU上，不会造成死锁。

破坏模块化

如果一个模块m1中方法g调用了另一个模块m2中的方法f，那么m1中的方法g需要知道m2的方法f使用了哪些锁。因为如果m2使用了一些锁，那么m1的方法g必须集合f和g中的锁，并形成一个全局的锁的排序。这意味着在m2中的锁必须对m1可见，这样m1才能以恰当的方法调用m2。

但是这样又违背了代码抽象的原则。在完美的情况下，代码抽象要求m1完全不知道m2是如何实现的。但是在具体实现中，m2内部的锁需要泄露给m1，这样m1才能完成全局锁排序。

性能

多核CPU下，想要发挥多核的性能不能使用粗粒度的big lock，而是要增加锁的数量。这势必导致代码的重构工作复杂。

自旋锁的实现

acquire接口

这里利用了RISC-V硬件提供的amoswap（atomic memory swap）原子指令。这个指令接收3个参数，分别是address，寄存器r1，寄存器r2。这条指令会先锁定住address，将address中的数据保存在一个临时变量中（tmp），之后将r1中的数据写入到地址中，之后再将保存在临时变量中的数据写入到r2中，最后再对于地址解锁。

注意：

  1. 原子指令的使用，不能用store等指令替代
  2. 在acquire这里首先禁止了中断（针对后续产生的UART中断）。spinlock需要处理两类并发，一类是不同CPU之间的并发，一类是相同CPU上中断和普通程序之间的并发。针对后一种情况，我们需要在acquire中关闭中断。中断会在release的结束位置再次打开，因为在这个位置才能再次安全的接收中断
  3. memory ordering。注意到接下来（包括acquire接口）的release接口都有一个__sync_synchronize()函数的调用。对于synchronize指令，任何在它之前的load/store指令，都不能移动到它之后。这意味着在释放锁之前，所有的临界区（critical section）操作对于其他CPU是透明（visible）的。

release接口

严格的在设置lk->locked之前让acquire调用push_off是很重要的。如果两者颠倒，会存在一个既持有锁又启用了中断的短暂窗口期，不幸的话定时器中断会使系统死锁。同样，只有在释放锁之后，release才调用pop_off也是很重要的。

睡眠锁

在等待获取锁时让出CPU，并允许在持有锁时让步（以及中断）。

因为睡眠锁保持中断使能，所以它们不能用在中断处理程序中。因为acquiresleep可能会让出CPU，所以睡眠锁不能在自旋锁临界区域中使用（尽管自旋锁可以在睡眠锁临界区域中使用）。

因为等待会浪费CPU时间，所以自旋锁最适合短的临界区域；睡眠锁对于冗长的操作效果很好。

CPU的排序规则称为内存模型（memory model）。

golang语言的内存模型为顺序一致性内存。

XV6中为了避免编译器的指令重排，在acquire和release中设置了__sync_synchronize()。该函数是一个内存障碍：它告诉编译器和CPU不要跨障碍重新排序load或store指令。

多线程

线程

XV6中，每个用户进程只拥有一个用户线程和一个内核线程。不同进程的用户线程有独立的地址空间，但内核线程对应的内核内存是共享的。

而在Linux等复杂的系统中，每个进程有多个线程。

线程调度

XV6为每个CPU核都创建了一个线程调度器（Scheduler），每一个调度器线程都有自己独立的栈。

内核出让cpu的两种情况：定时器中断与等待I/O设备

处理运算密集型线程：利用定时器中断。在每个CPU核上，都存在一个硬件设备，它会定时产生中断。所有的操作系统都会将这个中断传输到了内核，由此将程序运行的控制权从用户空间代码切换到内核中的中断处理程序（中断处理程序优先级更高）。而中断handler会自愿的将CPU出让（yield）给线程调度器。

以上这种方式称为抢占式调度（pre-emptive scheduling）。

具体实现可总结为：定时器中断会强制的将CPU控制权从用户进程给到内核，这里是抢占式调度；内核中用户进程对应的内核线程会代表用户进程出让CPU，这里是非抢占式调度。

其中用户寄存器存在trapframe中，内核线程的寄存器存在context对象中。同时要区分，调度器线程的context是保存在cpu结构体中，用户进程的内核线程的context保存在进程proc结构体中。所有其他在内存中的数据会保存在内存中不被改变，所以不用特意保存并恢复。

定时器中断具体代码流程：

yield()获取进程的lock，设置RUNABLE标志->sched()检查工作->swtch汇编，保存当前线程寄存器（callee需要保存的，相当于一次普通函数调用）->回到调度器线程所在的scheduler函数，设置c->proc=0 ，释放刚退出的进程的锁

注：

  • 区分不同的内核线程：myproc函数会使用tp寄存器来获取当前的CPU核的ID，并使用这个ID在一个保存了所有CPU上运行的进程的结构体数组中，找到对应的proc结构体。

  • 线程切换对锁的要求：线程切换过程必须在被切换线程中获取进程的锁，在调度器线程中释放该锁；禁止在调用switch时持有除进程自身锁（注，也就是p->lock）以外的其他锁。

  • swtch保存的是被调用方保存的寄存器（callee-saved registers）中。比较重要的是ra：保存了函数的返回地址；sp栈指针。

  • xv6启动时第一个调用swtch的调用者为allocproc函数（“假”的进程）

线程状态

XV6中线程的三种状态：

  1. RUNNING，线程当前正在某个CPU上运行
  2. RUNABLE，线程还没有在某个CPU上运行，但是一旦有空闲的CPU就可以运行
  3. SLEEPING，这节课我们不会介绍，下节课会重点介绍，这个状态意味着线程在等待一些I/O事件，它只会在I/O事件发生了之后运行

Sleep与Wakeup（以uart驱动为例)

Lost Wakeup：如果在调用sleep时，不释放uart的读写锁的话，后续urat中断将无法获取锁，但如果不在sleep函数的内部释放，那么在release和sleep之间可能发生中断，uart中断会wakeup应该会等待的一个读进程，但此时因为对应的sleep还未调用
